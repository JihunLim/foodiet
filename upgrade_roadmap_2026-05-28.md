### 1.3 백엔드 (Supabase)
- 15 마이그레이션, 6 Edge Functions(전부 production 상태):
  - `analyze-entry` · `generate-coach` · `generate-meal-plan` · `home-coach` · `community-report` · `render-share`
- Realtime 퍼블리케이션: `entries`, `community_posts`, `post_reactions`, `post_tips`, `group_invites`
- 보안 패턴 성숙: `SECURITY DEFINER` + `search_path` 핀, `service_role` 만 sanctions 접근, scope_json 트리거.

### 1.4 운영 모드 시그널 (중요)
최근 6 커밋이 전부 **운영 성숙도** 작업이다 — Android 빌드 통과 → Firebase → 서명 → Crashlytics → 보안 → 어댑티브 아이콘. 이는 다음 분기를 신기능 4주 프로젝트로 끌고가기보다는, **인프라를 재활용한 작고 누적 가능한 가치 증가**가 어울린다는 시그널이다.

## 2. 사용자 가치 갭 — 누가 무엇 때문에 끊기는가

> 출처: 기획안 §2 페르소나 + 현재 구현된 화면을 따라가며 "어디서 사용자가 잠시 멈출까" 추적한 결과.

### 갭 A — "어제 먹은 거랑 똑같은데 또 찍어야 해?"
- 페르소나 A(직장인 수민) 패턴: 출근길 아메리카노 + 점심 회사 식당 백반.
- 현재: 매번 카메라로 새로 촬영 + 분석 대기.
- 인사이트의 **"자주 먹은 음식 Top 5"** 데이터는 이미 있는데 → 다시 기록 액션과 연결 안 됨. **즐겨찾기/1탭 재기록 부재**.

### 갭 B — "3일 빠졌어. 이제 굳이?" (D30 이탈 모먼트)
- 현재: 끊김 감지는 있지만 (12시간 무기록 → 부드러운 리마인드 1회), **돌아왔을 때의 환영/회복 메커니즘 없음**.
- 단순 streak 카운트(`currentStreak`/`bestStreak`)는 있지만 **streak freeze 같은 회복 안전장치 부재**.
- 결과: D14 ~ D30 구간 이탈 가속.

### 갭 C — "활동량 슬라이더는 한 번 정하고 끝"
- 현재: 온보딩의 활동도 1~5 → 일일 칼로리 목표 고정.
- 실제 사용자의 운동/활동량은 **요일·주말·생리주기에 따라 다름**. 헬스킷 데이터 없이는 칼로리 모델이 늘 +/- 15% 오차.
- 페르소나 B(PT 재호)에게 특히 큰 갭 — PT 받는 날과 안 받는 날의 가용 칼로리가 다른데 앱은 이를 모름.

### 갭 D — "위젯이 예쁘긴 한데 보기만 해" 
- iOS 위젯 3종이 다 **읽기 전용** + 딥링크 (탭하면 앱 열림).
- iOS 17+ 부터 위젯에서 직접 액션 가능 (App Intents). Android 도 RemoteViews 액션 가능.
- **"한 잔 물 마셨어"** 한 탭 (앱 진입 없이) = 기록 마찰의 거의 0.

### 갭 E — "푸디는 좋아하는데 일방적이야"
- 푸디는 카드와 위젯에서 **한 줄 코멘트**만 보냄. 사용자가 묻고 답을 받는 양방향 채널 없음.
- 페르소나 A: "오늘 저녁 뭐 먹지?" 같은 질문이 가장 잦은데, 앱에서는 답을 받을 길이 없음 → 다른 앱(챗지피티/카톡채널)으로 이탈.

### 갭 F — "한식·배달이 좀 안 맞아"
- 한식·편의점·프랜차이즈 상위 500 자체 큐레이션은 있지만 (§18.1 #9), **배달앱 메뉴**(BBQ 황금올리브, 교촌 허니콤보 같은 SKU)는 LLM 추정 의존 → 신뢰도 낮음.
- 메뉴 DB 확장은 비용이 크지만 한국 시장의 핵심 경쟁력.
- **대응 위치**: §5 [V5] 로 별도 검증 항목 추가 (배달 상위 50 메뉴 LLM 정확도 측정 → 자체 DB 큐레이션 여부 결정). 또한 [S2] 푸디 채팅이 출시되면 "이거 무슨 음식인지 직접 물어보면 답해주는" 채널이 생겨 일부 우회 해결됨.

### 갭 G — "친구는 응원해주는데 같이 도전할 게 없네"
- 커뮤니티는 풍부하지만 **공동 목표 / 챌린지**가 없다. "이번 주 우리 다 5일 달성" 같은 미션 없음.
- 커뮤니티 기획서 Phase 3 항목인데 미구현.

## 3. 추천 업그레이드 — Quick Wins (1주 이내)

### [Q1] 위젯 인터랙티브화 — App Intents + 빠른 액션 🔥 추천 #1
**무엇이 바뀌나**
- iOS 17+: 위젯에 "🍓 사진 찍기", "💧 물 +1잔", "🥗 마지막 점심 다시 기록" 같은 **앱 진입 없는 한 탭 액션**.
- iOS 16 이하 / Android: 기존 딥링크 유지하되 "마지막 음식 즐겨찾기 재기록" 같은 1탭 인텐트 추가.

**왜 W1R 에 영향**
- 기록 마찰의 절반이 "앱 진입 → FAB 탭 → 카메라 권한 → 촬영"의 4단계.
- 위젯 인텐트는 이걸 **0단계**로 만든다. 잠금화면 위젯 + Apple Intelligence/Siri shortcut 까지 자연스럽게 연결.

**시작 전 확인**: Firebase Analytics 의 OS 분포로 iOS 17+ 비중과 위젯 사용률 확인 후, iOS 17 미만 대상 fallback 우선순위 결정 (현재 딥링크 유지가 디폴트).

**작업 규모**: 1주
- iOS: `AppIntent` 프로토콜 채택, 기존 `FoodietWidgetService.sync` 데이터 모델 확장.
- Android: `RemoteViews` + `PendingIntent.getBroadcast` + `BroadcastReceiver`. 이미 인프라가 있음.
- 백엔드 변경 없음 (`entries` INSERT + Realtime).

**의존성 / 리스크**
- iOS 17 미만 사용자는 fallback 필요 (딥링크 유지).
- "마지막 음식 재기록" 의도가 의도치 않게 발화 안 되도록 confirm 다이얼로그.

**참고 파일**: `app/ios/FoodietWidgets/`, `app/android/.../widget/FoodietWidgets.kt`, `app/lib/services/home_widget_service.dart`.

### [Q2] 자주 먹는 음식 즐겨찾기 + 1탭 재기록 🔥 추천 #2
**무엇이 바뀌나**
- 인사이트의 `_TopFoodsCard` 항목에 **별표** 추가 → "즐겨찾기"에 고정.
- 홈 화면 카메라 FAB 위에 **"빠른 기록"** 칩 행 (즐겨찾기 4~5개 가로 스크롤).
- 탭 → 사진 없이 entry 생성 (`source = 'favorite'`), 직전 같은 음식의 매크로 평균 자동 적용.

**왜 W1R 에 영향**
- 페르소나 A 의 루틴 식단(아메리카노/회사 식당/편의점 도시락) 케이스에 정확히 적합.
- "사진 한 장도 귀찮은 순간"에 마찰이 0이 됨. 다이어트 앱 이탈의 가장 큰 트리거 제거.

**작업 규모**: 1주
- DB: `entry_items` 에서 사용자별 `name` 카운트 집계 (이미 SQL 존재). 즐겨찾기 플래그용 `favorites` 테이블 신규 (`user_id`, `name`, `template_macros jsonb`, `pinned_at`).
- 클라: `entries_provider` 에 `createFromFavorite()` 추가 + 홈 위 칩 UI.

**의존성 / 리스크**
- "사진 없는 entry" 가 신뢰도 0.99 로 기록되면 분석 통계가 왜곡될 수 있음 → 별도 `source` 값으로 인사이트에서 분리 표기.
- 양 조절 UX (어제와 같은 양 이라고 가정) — 첫 사용 시 +/- 한 번만 묻기.

### [Q3] 연속 기록 보호 + 끊김 회복 코칭 ⭐ D30 직접 영향
**무엇이 바뀌나**
- **Streak Freeze**: 주 1회 자동 적립. 무기록 날 1일까지 streak 보존 (Duolingo 모델).
- **돌아오는 날 환영**: 3일+ 무기록 후 첫 기록 시 푸디가 "다시 와줘서 고마워. 천천히 가자" 식으로 **체중·칼로리 압박 없는 부드러운 onboarding-back**.
- 인사이트 헤로 스탯(`_HeroCard` 의 "현재 연속") 옆에 streak freeze 잔량 표시 ("🛡 1주 안에 1번 빠져도 OK").

**왜 D30 에 영향**
- D14 ~ D30 이탈의 1차 원인 = "한번 끊기면 다시 시작 부담". streak freeze 는 이 부담을 구조적으로 제거.
- "돌아온 사용자"는 보통 D60 까지 살아남음 — 1주에 1번 회복하는 사용자는 사실상 활성 유저.

**작업 규모**: 3~5일
- DB: `profiles` 에 `freeze_remaining int default 1`, `freeze_renewed_at date`. 주간 cron 으로 갱신.
- 클라: `home_today_page` 의 "현재 연속" 카드 옆에 방패 UI, 푸디 messaging branch.

### [Q4] 수분 알림 + 위젯에서 물 1탭 기록
**무엇이 바뀌나**

## 5. 추천 업그레이드 — Validate-First (검증 후 결정)

> "지금 시작하면 안 된다"는 뜻이 아니라, 작은 실험으로 가치 확인 후 본 작업 들어가는 게 ROI 가 높다는 항목들.

### [V1] Pro 구독 출시 (기획안 §17)
**가설**: 광고 제거 + 식단 추천 무제한 + 푸디 챗 무제한 + 심화 인사이트 = 월 ₩4,900 / 연 ₩39,000.

**먼저 검증할 것**
1. **AdMob 인사이트 광고의 IAA (광고 거부감) 측정** — 인사이트 탭 이탈률 vs 비광고 대조군 A/B (App Store 가이드 허용 범위 내).
2. **푸디 챗 ([S2]) 출시 후 첫 4주 일 평균 사용 메시지 수** — 무제한 베타로 운영 후 분포 측정.
3. **목표 사용자 수**: 결제 전환 5% 가정 시 MAU 2,000명 = 월 100명 × 4,900 = ~50만원. **BEP 가 너무 낮으면 광고 모델이 낫다**.

**상황 시그널**: 솔로 개발 + 사용자 베이스 확립 전 단계. **3 ~ 6개월 후 재평가** 가 합리적.

## 8. 의도적으로 제외한 항목 (왜 추천하지 않는가)

> 신뢰도 시그널 — "떠올랐지만 빼는 이유"가 추천 항목의 정당성을 만든다.

| 후보 | 제외 이유 |
|---|---|
| **바코드 스캔으로 가공식품 영양 정보** | 사진 분석(`analyze-entry`)이 이미 일반화돼 있어 ROI 낮음. 편의점 도시락은 [S1]/[V1] 후 제휴 DB 로 푸는 게 자연스러움. |
| **OCR 영수증** | 사진 한 장으로 끝나는 모델과 충돌 — "한 장 더 찍어야 함"이 본질 가치를 약화. |
| **레벨 / 배지 / 랭킹** | 커뮤니티 기획서 §1.3 "비교하지 않는다" 원칙 위반. 챌린지([S3])로 대체. |

| **위치 기반 식당 추천** | 외부 DB 의존(카카오/네이버 플레이스 API), 광고로 보이기 쉬움. 푸디 챗([S2])에서 자연스럽게 답할 수 있는 영역. |
| **음식 사진 갤러리 (인스타 스타일)** | "다이어트 앱"이 아니라 "푸드 인스타"로 포지셔닝 표류. |

## 10. 결정 필요 항목 (이 리포트 채택 시)
